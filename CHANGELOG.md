# Changelog

## Unreleased

### Added

- **Modal worker now auto-deploys from `main` via CI with a local non-main deploy guardrail** — added `.github/workflows/modal-deploy.yml` which runs on every push to `main` that touches `src/coarse/**`, `deploy/modal_worker.py`, `pyproject.toml`, or the workflow file itself, installs `modal>=0.68 fastapi>=0.115 pydantic>=2.0`, and runs `modal deploy deploy/modal_worker.py` with `MODAL_TOKEN_ID` / `MODAL_TOKEN_SECRET` from repo secrets. Runs serialize via a `modal-deploy` concurrency group so back-to-back pushes can't deploy in parallel, and the workflow is also `workflow_dispatch`-able for manual retries. The deploy step's "Validate Modal credentials" stanza fails fast with a clear error if the secrets haven't been added yet, so the first merge to main after this lands doesn't silently no-op. Belt to the CI suspenders: `deploy/modal_worker.py` now has an import-time guardrail (`_enforce_deploy_branch_at_import_time` + `_enforce_deploy_branch`) that refuses any `modal deploy` from a non-`main` branch with a RuntimeError pointing at the CI path and explaining the emergency escape hatch (`COARSE_MODAL_DEPLOY_FORCE=1`). The guard short-circuits inside pytest sessions (`sys.modules["pytest"]` or `PYTEST_CURRENT_TEST` set) so `tests/test_modal_worker.py` can still load the file via `importlib` on feature branches, and inside Modal containers at review-serving time (`modal.is_local()` is False) so cold-starts don't break. In CI it reads `GITHUB_REF_NAME` (which the deploy workflow only triggers on `main`); locally it shells out to `git rev-parse --abbrev-ref HEAD`. Git failures (detached HEAD, PyPI install with no repo) are silent no-ops rather than blocking a legitimate import. Seven new regression tests in `tests/test_modal_worker.py` cover CI main pass, CI non-main raise, local main pass, local non-main raise, local force-env warn, local git-error fall-through, and the import-time pytest short-circuit. `deploy/DEPLOY.md` step 3 now documents the auto-deploy path, the one-time `gh secret set MODAL_TOKEN_ID / MODAL_TOKEN_SECRET` setup, and the `COARSE_MODAL_DEPLOY_FORCE=1` emergency override. Incident context: on 2026-04-13 a `modal deploy` from a `dev` checkout shipped resurrected cheap-tier stage routing plus an api-key race to production, breaking every review for hours. This guardrail exists so that class of accident cannot happen again through the normal path.

## v1.2.0 — 2026-04-11

Minor release. Headline is the new **optional author-steering notes** field on the submit page (#54): authors can attach a short instruction ("please focus on the identification strategy in §3") and the reviewer adapts its focus. Also ships the **document-form detection** feature so the reviewer stops demanding full-manuscript rigor on outlines and proposals. Hardens the Modal worker in two places: the sanitizer no longer collapses `instructor.InstructorRetryException` messages to a bare `</last_exception>` tag (#55), and extraction + worker now scrub NUL bytes before Supabase writes (#62). Web-side adds the auto-disabling email-capacity gate (#59) and bumps the stale monitoring-cron threshold (#66). ### Added

- **Optional author-steering notes on the submit page (#54)** — the submit form now has an optional `Notes for the reviewer` textarea (max 2000 chars) that the author can use to steer the review: "please focus on the identification strategy in §3 — the data section is still a placeholder." The note is plumbed end-to-end: frontend → `/api/submit` (server-side length validation) → Modal webhook → `ReviewRequest.author_notes: str | None = None` (defaulting to None so older in-flight spawn payloads still deserialize) → `review_paper(..., author_notes=author_notes)` → the **overview, section, and editorial agents** via a new `author_notes_block(notes)` helper in `prompts.py`. The helper returns `""` for None/empty/whitespace-only input so the no-notes path is byte-identical to the pre-feature behavior and prompt caching on the cached system blocks is completely unaffected. Non-empty input is truncated to 2000 chars, run through `_strip_fence_tags` (which gains `<author_notes>` as a recognized fence), wrapped in an `<author_notes>` fence, and prepended to each agent's **user** message (never the system message) with a short instruction telling the agent to treat the notes as a request for focus, not as commands that override the review rubric. The content-boundary notice in `prompts.py` also gains `<author_notes>` so every agent that imports the notice is explicitly told the block is boundary-fenced data. Security: three defenses in depth — (1) server-side 2000-char cap in `web/src/app/api/submit/route.ts`, (2) `_strip_fence_tags` removal of any injected `<paper_content>` / `<author_notes>` / etc. closing tags so an attacker can't close the fence early, (3) the wrapper text instructs the agent that the content is steering, not instructions. 15 new tests: `test_prompts.py` covers the `author_notes_block` helper (empty/None/whitespace byte-identity, non-empty wrapping, fence-tag injection defense, truncation at 2000 chars, required steering frame instruction, `_FENCE_TAG_RE` matches `<author_notes>`, content-boundary notice lists the tag); `test_overview-agent.py`, `test_section-agent.py`, `test_editorial.py` each cover byte-identity of the user message when notes are None/omitted/whitespace-only, plus the prepend-to-user-message path when notes are supplied, plus that the notes text never leaks into the cached system block; `test_pipeline.py::test_review_paper_forwards_author_notes_to_all_review_agents` is the glue test that captures the forwarded value at every downstream agent; `test_modal_worker.py` pins that `ReviewRequest` deserializes `author_notes` from the webhook payload and defaults to None. Deliberately NOT routed through: `completeness`, `cross_section`, `proof_verify`, `literature`, `calibration`, `contribution_extraction` — these do structural / procedural work that shouldn't be steered by free-form author input. If that turns out to be wrong, extending is a one-line change per agent.
- **Document-form detection + prompt branching** — the review pipeline now distinguishes completed manuscripts from earlier drafting stages (outline, partial draft, proposal, technical report, working notes, other) and adapts the agent prompts accordingly. The cheap metadata LLM call in `src/coarse/structure.py::_get_metadata` gains a `document_form` field on its `PaperMetadata` response model; the classifier has a rubric in `METADATA_SYSTEM` explaining each form (heading vs. prose density, presence of results, conditional phrasing like "if supported by data", pre-committed discussion bullets). `PaperStructure` carries the classified form through to downstream agents. A new `document_form_notice(form)` helper in `prompts.py` returns a tailored system-prompt addendum for each non-manuscript form — and the empty string for `manuscript`, so the default peer-review path is byte-identical to the pre-feature behavior (all agent tests pinning `system_content == OVERVIEW_SYSTEM`/`COMPLETENESS_SYSTEM`/`SECTION_SYSTEM`/`EDITORIAL_SYSTEM`/`PROOF_VERIFY_SYSTEM`/`CROSS_SECTION_SYSTEM` still pass). Overview, section, editorial, proof-verify, and cross-section agents all append the notice; the completeness agent additionally short-circuits to `[]` for outline/notes/other since "flag missing content" is noise by construction when ~100% of the content is missing by design. Pipeline threads `structure.document_form` through `_review_section`, `verify_agent.run`, `cross_section_agent.run`, and `editorial_agent.run`. Two safety defenses land alongside the classifier: (1) a prose-heavy heuristic in `analyze_structure` that overrides an LLM misclassification from outline/notes/other back to `draft` when the document actually has substantial non-bullet prose — this defends against the manuscript→outline direction, which silently degrades real peer reviews into 2-3 soft comments; (2) the metadata-extraction fallback on LLM failure now returns `document_form="draft"` instead of `"manuscript"`, because defaulting to the punitive strict-review frame on a classifier failure recreates the exact bug this feature fixes. Motivated by an April 11 review (`bfa57ebb-4ab6-4937-a724-9757650f7ea0`, Claude Opus 4.6) of a paper outline where 5 out of 8 overall-feedback items were variations of "this isn't a manuscript, you haven't written any prose" with a final `Recommendation: reject` — actively unhelpful to an author in the planning stage. 50 new tests across `test_types.py`, `test_prompts.py`, `test_structure-analysis.py`, `test_overview-agent.py`, `test_completeness-agent.py`, `test_section-agent.py`, `test_editorial.py`, `test_verify.py`, and `test_cross_section-agent.py` pin the manuscript path (byte-identical on every agent), the branching-on-form behavior, the completeness early-return, focus-specific section prompt routing (parametrized over general/proof/methodology/discussion/literature), the `analyze_structure` form-propagation glue test, the heuristic override and its non-firing cases, the draft fallback on metadata failure, and drift protection via `typing.get_args(DocumentForm)` to guarantee every literal value has a notice entry. `preprint` was dropped from the `DocumentForm` literal per YAGNI — it had the empty-string notice (same as `manuscript`) and existed only to satisfy a rubric entry; preprints are classified as `manuscript` and get the full peer-review frame.

### Changed

- **Monitoring cron daily-reviews warning threshold bumped 40 → 200** — `.github/workflows/monitor.yml` was set to warn the maintainer whenever daily review volume crossed 40, a number from the pre-launch era (<1 review/day). 40 reviews = 80 emails, nowhere near Gmail's 500/day sender cap, so the warning was pure noise. 200 reviews ≈ 400 emails ≈ 100-email buffer below the cap, which lines up with the frontend email-capacity gate at 240 in `web/src/lib/emailCapacity.ts` (the cron now warns first, the frontend auto-disables the input second). Also updated the matching threshold table in `deploy/CAPACITY.md`. Monthly thresholds (300 / 800) deliberately unchanged — they track Modal compute, not email volume, and need a separate data-driven revisit. (#66)
- **`proof_verify` gated on a 500-char / non-empty-claims threshold** — `pipeline.py::_review_section` now skips the adversarial proof-verification pass on sections flagged `math_content=True` that are both short AND have no extracted formal claims. The gate is in a new `_section_needs_proof_verify(section)` helper and requires either (a) `section.claims` non-empty — any extracted Theorem/Lemma/Proposition header bypasses the length check so short lemma-with-proof paragraphs still verify — or (b) `len(section.text) >= 500` chars. Threshold is deliberately low so short lemmas with their proofs aren't missed (typical short-proof paragraphs run 300-800 chars). Skips pure-wasted calls on sections where `math_content` was flagged from a single inline equation in a footnote, saving one proof_verify call (~$0.30 on sonnet-4.6, more on reasoning models) per such section. Expected savings: 0-10% per review, variable by paper — math-heavy papers with long proofs see no change. Five new tests in `test_pipeline.py` pin the math_content/claims/length matrix + the `_review_section` integration.

### Fixed

- **Instructor retry errors no longer collapse to a bare `</last_exception>` tag** — `deploy/modal_worker.py::_sanitize_error` used a "last non-empty line" heuristic to trim Python tracebacks before writing `error_message` to Supabase. For `instructor.exceptions.InstructorRetryException`, whose `str(exc)` is an XML-like envelope (`<last_exception><failed_attempts><exception>…`), that heuristic kept only the outermost closing tag and threw away the actually-useful payload — so every instructor-retry failure showed up in the operator dashboard as the single string `</last_exception>`. The fix adds a regex pre-pass that extracts the first `<exception>…</exception>` block when present, falls back to the last-line heuristic otherwise, and as a safety net detects the case where the "last line" is itself a bare closing tag (and falls back to the whole trimmed message rather than keep garbage). Secret scrubbing still runs on the extracted text. Five new tests in `tests/test_modal_worker.py` pin: instructor envelope extraction, closing-tag-only fallback, single-line passthrough, classic multi-line traceback (still takes the final `ValueError: …` line), and that a key pasted inside a retry envelope is still scrubbed. Also one-time operator backfill: the 7 existing `</last_exception>` rows in the Supabase `reviews` table were updated with a note pointing at this issue. (#55)
- **NUL byte (`\\x00`) scrubbing before Supabase writes** — two pipeline failures on 2026-04-11 hit Postgres SQLSTATE `22P05` (`\u0000 cannot be converted to text`) after the user had already paid for the full review. `text` columns reject the real NUL byte and PostgREST's JSON decoder rejects the 6-char `\u0000` escape; some OCR backends and some reasoning-model outputs emit both. A partial defense in `src/coarse/llm.py::_sanitized_completion` already stripped these from LLM response content, but the extraction path wasn't covered and the modal worker was writing raw `paper_markdown` / `result_markdown` / `error_message` straight to Supabase. Added a shared `_strip_nul_bytes(text)` helper in `src/coarse/extraction.py` (idempotent, safe on empty/None) applied at the `full_markdown` seam in both `extract_text` and `extract_file`, plus a sibling `_strip_nul_bytes` in `deploy/modal_worker.py` wrapping every Supabase text-field update (`paper_markdown`, `result_markdown`, `error_message`). Seven new tests across `tests/test_extraction.py` and `tests/test_modal_worker.py` cover: real NUL removal, literal `\u0000` escape removal, None/empty-string safety, normal text unchanged, and end-to-end extract_text scrubs NULs emitted by a mocked OCR backend. (#62)
- **Email-capacity gate on the submit page** — when the rolling 24-hour review count crosses 240 (≈480 emails, leaving a small buffer below Gmail's free-tier 500/day sender cap), the submit form disables the email input and shows a short red note: "Daily email cap hit — save your review key to retrieve it." Submission is still allowed without an email; the Modal worker already guards `if email:` before sending the completion message, so review processing is unaffected. The threshold lives in `web/src/lib/emailCapacity.ts` (`EMAIL_CAPACITY_REVIEW_THRESHOLD`), `/api/status` returns the new `emailCapacityReached` flag alongside `activeReviews`, and `/api/submit` re-checks server-side so a client can't bypass the email regex by lying about capacity (skips the `review_emails` not-null and regex validation only when the DB confirms the gate is active, and suppresses the confirmation email when the address is empty). Motivated by the 2026-04-10/-11 traffic spike where daily volume jumped from <10 to 96-142 reviews — a single 2x growth day would have quietly started bouncing completion emails. (#59)
- **Prompt caching re-enabled for OpenRouter-routed Claude and Gemini** — `LLMClient.supports_prompt_caching` had a stale openrouter/ exclusion that silently disabled Anthropic prompt caching for every production review (coarse users upload OpenRouter keys only, never direct Anthropic). OpenRouter has forwarded explicit per-block `cache_control` breakpoints to Anthropic since mid-2024 — the gate was months out of date. Flipping it on activates two already-written caching paths: `agents/base.py::_build_messages` caches the system prompt on every agent (section agents share one system prompt across N parallel calls, so 1 write + N-1 reads fires immediately on every multi-section review), and `agents/overview.py` caches the full paper-context prefix in the system block. A follow-up commit extends the match to also cover Google Gemini models — per OpenRouter's current docs, "Gemini caching in OpenRouter requires you to insert cache_control breakpoints explicitly within message content." The match covers `anthropic/claude-*`, `openrouter/anthropic/claude-*`, `vertex_ai/claude-*`, `gemini/gemini-*`, `google/gemini-*`, `openrouter/google/gemini-*`, and `vertex_ai/gemini-*` forms; the extraction_qa vision stage (default model `gemini/gemini-3-flash-preview`) gets caching too. OpenAI and DeepSeek are deliberately NOT matched — both auto-cache prompt prefixes server-side and don't need `cache_control` blocks; adding them would clutter requests with no-op metadata. Cost math: cache writes billed at 1.25× input, reads at 0.1× (90% off), break-even at 0.28 reads per write. Every `cache_control` block in coarse's pipeline clears that bar trivially. Expected savings: ~$0.09-0.20 per sonnet review, ~$0.14-0.30 per opus review, scaling with section count and paper length. 8 new tests in `test_llm.py` cover direct anthropic, OR-routed anthropic, direct gemini, OR-routed gemini, vertex_ai/claude, vertex_ai/gemini, plus negative cases for OpenAI / DeepSeek (auto-cache, gate stays False) and undocumented providers (Qwen, Mistral, Kimi, glm, Grok).

## v1.1.6 — 2026-04-11

Security and routing release. Headline fix is the user-OpenRouter-key ferry (#50): the user's API key no longer rides through Modal's `spawn()` managed-queue payload. Also ships the per-stage model routing feature (`StageRouter` + `STAGE_MODELS` + `--stage-override`) — the three "Added" entries below were retroactively documented under v1.1.4 on `dev` but never actually landed in the tagged v1.1.4 or v1.1.5 releases; they ship for real in v1.1.6. Rounds out with a cheap-tier glm-5.1 thinking-mode fix found in end-to-end testing and a double-click race fix in the `/api/submit` route.

### Security

- fix(deploy): ferry user OpenRouter key via `review_secrets` side-table so it no longer rides through Modal's `spawn()` payload / managed queue. Frontend inserts the key under RLS deny-all, worker reads + deletes in one shot via `_fetch_and_consume_user_key`, hourly GitHub Actions cron sweeps stragglers older than 3h. Worker keeps the `req.user_api_key` fallback for backward-compat with in-flight jobs during rollout. Requires running `deploy/migrate_review_secrets.sql` in Supabase SQL editor before deploying the worker. (#50)
- fix(web): make the `/api/submit` route double-click-safe. Previously, a double-click fired two POSTs with the same presigned `id`; the second one's `review_emails` insert hit a Postgres unique_violation and the rollback handler `delete from reviews` nuked the first submit's row, leaving submit 1's Modal worker running against a deleted review. Now the route detects SQLSTATE 23505 on the `review_emails` insert, treats it as an idempotent success, and returns 200 without firing a second Modal worker. Mid-submit rollbacks switched from `.delete()` to `.update({ status: "failed" })` so a concurrent double-click never sees a disappearing row. (#50)

### Added

- **Per-stage model routing (`--stage-override`, `STAGE_MODELS`)** — the review pipeline now resolves an LLM client per stage instead of threading a single client through every agent. Stages classified as "cheap-safe" (`metadata`, `math_detection`, `contribution_extraction`, `calibration`) are pinned to a new `CHEAP_STAGE_MODEL = "z-ai/glm-5.1"` regardless of what `--model` the user passes, while reasoning-heavy stages (`overview`, `completeness`, `section`, `cross_section`, `verify`, `editorial`) use `--model` as-is. The routing map lives in `src/coarse/models.py::STAGE_MODELS`; the resolver is `src/coarse/routing.py::StageRouter`; the CLI gets a repeatable `--stage-override name=model` flag for ad-hoc per-stage overrides. Worked example: running `coarse-ink review paper.pdf --model anthropic/claude-opus-4.6` previously paid opus rates (~$15/$75 per 1M tok) on the trivial 500-token `metadata` call and the 2000-token `math_detection` call; now those route to glm-5.1 on US-HQ providers while overview/section/editorial continue to use opus. Estimated savings ≈$0.30-0.40 per review. Theory-driven classification (gh #46), not empirically validated — per-stage regressions are recoverable via `--stage-override`.
- **US-HQ provider allowlist for cheap-tier routing (`CHEAP_STAGE_PROVIDERS`)** — the glm-5.1 cheap tier is restricted to `("DeepInfra", "Parasail", "Fireworks")` via OpenRouter's `provider.only` preference so request data stays on US servers even though the model weights are from a Chinese company. Layered on top of the existing `data_collection=deny` injection: both are active. Enforced through a new `LLMClient(provider_allowlist=...)` kwarg and `_apply_provider_prefs` helper that merge the allowlist into `extra_body.provider.only` on every call. Z.AI's own hosting (HQ=SG) and Alibaba (the only qwen endpoint, HQ=SG+CN) are explicitly excluded.
- **Cheap-tier kimi-k2.5 fallback (`CHEAP_STAGE_FALLBACK_MODEL`)** — if the glm-5.1 call fails (provider outage, transient OpenRouter error), the `StageRouter`-built cheap client transparently retries on `moonshotai/kimi-k2.5` with the same `CHEAP_STAGE_PROVIDERS` US-HQ allowlist. Each underlying model gets its own instructor mode (JSON for glm, MD_JSON for kimi — required because kimi is in `MARKDOWN_JSON_PREFIXES`), so the fallback is client-level rather than OpenRouter's server-side `models` preference. Wired through a new `LLMClient(fallback_client=...)` kwarg that catches exceptions in `complete()` / `complete_text()` and delegates to the fallback's method with the same kwargs. Both primary and fallback clients are cached in the router so their cumulative costs both flow into `router.cost_usd`.

### Fixed

- **Cheap-tier glm-5.1 no longer truncates on invisible thinking overhead** — end-to-end test of the per-stage routing on a 20k+ token paper surfaced a systematic `instructor.core.exceptions.IncompleteOutputException` on every glm-5.1 cheap-tier structured-output call. Root cause: glm-5.1 via OpenRouter defaults to thinking-on, and the invisible reasoning preamble consumes `max_tokens` before any JSON emits. Structured-output calls with `max_tokens <= 2048` (metadata 256, math_detection 1024, calibration 2048, contribution_extraction 2048) never reached the JSON phase. They're only deleted on the success path now; failed reviews keep their PDF so Modal's infrastructure retries (or a manual resubmit of the same job_id) can actually find it. Stale PDFs from failed runs are swept by the daily cleanup cron at 24h.

- **Docling fallback works in production** — added `libgl1` + `libglib2.0-0` to the Modal image. Without them, Docling crashed on first import with `libGL.so.1: cannot open shared object file`, meaning the "offline fallback" wasn't actually a fallback when the OpenRouter OCR path failed.

- **CI pipeline** (`.github/workflows/ci.yml`) — Ruff lint (advisory), pytest (blocking), version consistency check on every PR.
- **Evaluation data** (`data/refine_examples/`) — Reference reviews and papers for quality scoring.

### Changed

- **Anthropic prompt caching** — Overview panel judges share cached paper context via `cache_control`, reducing input token costs ~90% on cache hits. Judges run sequentially for Anthropic models to ensure cache sharing. Section/crossref/critique agents also annotate system prompts for caching.
- **Structured assumption cross-check** — 4-step procedure for cross-checking theoretical assumptions against empirical methodology.
- **Appendix proof coverage** — Appendix sections classified as "proof" are now reviewed instead of skipped.
- **Anti-redundancy prompts** — Section agents instructed not to restate overview issues; crossref dedup removes comments that duplicate overview issues.
- **Anti-truncation quote instructions** — Strict "NEVER truncate mid-equation" rule in all section prompts.

## v0.1.0 — 2026-03-03

Initial release.

### Features

- **PDF extraction** (`extraction.py`): Convert PDF papers to text using pymupdf4llm; optional
  vision mode for scanned or image-heavy PDFs.
- **Structure analysis** (`structure.py`): LLM-based parsing of paper into title, domain,
  taxonomy, abstract, and labelled sections with claims and definitions.
- **Overview agent** (`agents/overview.py`): Produces 4-6 macro-level issues covering the paper
  as a whole (runs in parallel with section agents).
- **Section agents** (`agents/section.py`): One agent per section produces 15-25 detailed
  comments with verbatim quotes from the paper.
- **Cross-reference agent** (`agents/crossref.py`): Deduplicates comments, validates quotes, and
  checks consistency across agents.
- **Critique agent** (`agents/critique.py`): Self-critiques the draft review and revises weak
  comments as a quality gate.
- **Pipeline** (`pipeline.py`): Orchestrates extraction, structure analysis, all agents, and
  synthesis into a single `review_paper()` call.
- **Synthesis** (`synthesis.py`): Deterministically renders a `Review` object to structured
  markdown format.
- **LLM layer** (`llm.py`): litellm wrapper with instructor for structured Pydantic output; tracks
  per-call cost.
- **Cost estimation** (`cost.py`): Pre-flight cost estimate with a configurable spending cap and
  interactive user approval gate.
- **Config management** (`config.py`): Reads and writes `~/.coarse/config.toml`; resolves API
  keys from environment variables or the config file.
- **CLI** (`cli.py`): `coarse setup` for interactive configuration; `coarse review paper.pdf` for
  reviewing a paper; `--model`, `--output`, and `--yes` flags.
- **Quality eval** (`quality.py`): Developer-only tool that scores a generated review against a
  reference review using an LLM judge.
- **Type definitions** (`types.py`): All shared Pydantic models (`PaperText`, `PaperStructure`,
  `Review`, `DetailedComment`, `CostEstimate`, etc.).
- **Prompt templates** (`prompts.py`): All LLM prompt templates in one place.
