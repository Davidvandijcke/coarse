# Changelog

## Unreleased

### Added

- **Automated preview Supabase migration runner.** New `scripts/apply_migrations.sh` applies every `deploy/migrate_*.sql` exactly once per project, tracked in a `schema_migrations (name text primary key, applied_at timestamptz)` table the script creates on first use and reads on every subsequent run to skip already-applied files. New `.github/workflows/supabase-migrate-preview.yml` fires on any `dev` push that touches `deploy/migrate_*.sql`, `scripts/apply_migrations.sh`, or the workflow itself, installs `postgresql-client`, reads the `PREVIEW_SUPABASE_DB_URL` secret from the `preview-modal` GitHub Environment, and runs the script against preview Supabase before the paired `modal-preview-deploy.yml` run so preview Modal workers never boot against an unmigrated DB. New `deploy/bootstrap_schema_migrations.sql` is a one-shot the operator runs in the Supabase SQL editor once per project, seeding the tracking table with every historical migration already applied by hand during initial setup. This exists because several of those files are deliberately destructive (e.g. `migrate_email_phase2.sql` does `alter table reviews drop column email`) and would crash if replayed; after bootstrap, the tracking table guarantees they're skipped on every CI run while new migrations added later apply exactly once. Behavioral coverage via a mock-psql harness: fresh run applies all, second run skips all, dry-run shows pending without mutating state, failure mid-run stops the loop and exits 1 with the tracking table reflecting only pre-failure successes, and a bootstrap-seeded run applies only the one un-seeded migration. **Operator action required once:** run `deploy/bootstrap_schema_migrations.sql` in the preview Supabase SQL editor, then `gh secret set PREVIEW_SUPABASE_DB_URL --env preview-modal --repo Davidvandijcke/coarse --body "<connection-string>"` using the preview project's Connection String (URI tab) from Supabase → Settings → Database. Same steps apply to prod whenever a matching prod-migrate workflow is wired up.

- **`deploy/migrate_drop_mcp_handoff_tokens.sql` — clean up the retired `mcp_handoff_tokens` table** from any project that still has it lingering from the deleted `migrate_mcp_handoff.sql` (see below). Single statement: `drop table if exists mcp_handoff_tokens`. Idempotent, no-op on fresh projects, tracked in `schema_migrations` so it runs exactly once per existing DB.

### Fixed

- **`deploy/supabase_schema.sql` status constraint was inconsistent with the `count_active_submitted_reviews` RPC** that lives in the same file. The table definition restricted `reviews.status` to `(queued, running, done, failed, cancelled)`, but the RPC at the bottom of the file filters `r.status in ('queued', 'running', 'extracting', 'extracted')` — two statuses the check constraint would have rejected at insert time. Current preview and prod Supabase projects are unaffected because both were bootstrapped from the older baseline and then had `migrate_mcp_handoff.sql` applied (which widened the constraint to include the extracting/extracted statuses), but any fresh Supabase project bootstrapped from `supabase_schema.sql` in its current state would have failed the first time a paper was handed off to the CLI and transitioned to `status='extracting'`. Bug was introduced in `d0706b2 feat(release): retire MCP server path entirely (v1.3.0)` when the baseline was rebuilt but the widened constraint block was dropped without being re-inlined. Fix: baseline constraint now lists all seven current statuses (`queued, running, extracting, extracted, done, failed, cancelled`) so a fresh-project bootstrap matches the running preview/prod state.

### Removed

- **`deploy/migrate_mcp_handoff.sql`** — reaped as part of the MCP server retirement (`d0706b2`). The file created the now-dead `mcp_handoff_tokens` table along with three load-bearing operations (`review_handoff_secrets` table, `reviews.taxonomy` column, widened `reviews_status_check`) that the v1.3.0 retirement commit correctly inlined into `supabase_schema.sql` — with the exception of the widened status check (tracked as the bug above). With that baseline fix landed, the migration file has no remaining purpose for fresh bootstraps, and `deploy/migrate_drop_mcp_handoff_tokens.sql` now handles the dead-table cleanup on existing preview/prod projects via the new migration runner. `deploy/bootstrap_schema_migrations.sql`'s seed list was updated to drop `migrate_mcp_handoff` accordingly.

### Fixed

- **Headless CLI transport hardening round 2 — retry loop, model fallback, concurrency semaphore, skill version sentinel, large-prompt test.** Five MEDIUM/LOW-severity gaps from the post-probe integration audit, on top of the HIGH-severity round-1 fixes in commit edb6dde. (1) **Retry loop**: `_HeadlessCLIClient._run` now wraps `subprocess.run` in a 3-attempt retry with 2s/4s backoff between attempts. The new `_is_transient_cli_error(stderr, returncode)` classifier catches rate-limit, 429/502/503/504, connection reset, upstream overloaded, "please try again", and GNU timeout(1) exit 124 patterns; matching errors retry, everything else fails fast. Auth failures, `FileNotFoundError` (missing binary), Python-level timeouts, and generic permanent errors all skip the retry so we don't loop on truly broken calls. Previously a single transient rate-limit during a 20-section parallel review would kill the entire 25-minute pipeline — now the worst case is a 6-second detour per failing call. (2) **Unknown-model fallback**: `_is_unknown_model_error(stderr)` catches "unknown model" / "model not found" / "invalid model" patterns, and each client's new `_can_fall_back_to_default_model()` / `_fall_back_to_default_model()` hooks swap the user-provided model for the class default (`HEADLESS_DEFAULT_MODELS[host]`) and retry once. Instance-scoped `_model_fallback_attempted` flag caps the fallback at one shot so a broken default doesn't loop. Codex and Gemini skip the fallback when the user didn't provide an explicit model (they already default to the CLI's built-in choice). (3) **Concurrency semaphore**: each client class owns a `threading.Semaphore(N)` via the new `_HeadlessCLIClient._get_semaphore` classmethod, lazily initialised on first use with double-checked locking. Default `N=3`, overridable via the new `COARSE_HEADLESS_CONCURRENCY` env var (validated to 1-20, invalid values fall back to default with a warning). `_run` acquires the semaphore via `with type(self)._get_semaphore():` so at most 3 `claude -p` subprocesses run simultaneously even when the pipeline spawns 10 parallel section agents — prevents the "20-section paper trips Claude Max rate limit at section 11" failure mode. Per-class (not global) so mixed-host runs don't cross-contend. (4) **Skill version sentinel**: `coarse install-skills` now writes a `.coarse-version` file next to each `SKILL.md` containing `__version__`. A second install with a different version logs `✓ claude → <dest> (replaced 1.3.0 → 1.4.0)` so users know their local skill bundle was refreshed. First-install paths and identical-content paths stay silent. New `_read_skill_version(dest)` helper reads the sentinel without crashing on permission errors. (5) **Large-prompt regression test**: new `test_run_handles_large_prompt_stdin` pipes a 500 KB synthetic prompt through `_run` with a mocked `subprocess.run` that asserts the full input arrived — pins the current behaviour so a future switch away from `subprocess.run(..., input=prompt)` to explicit `Popen` can't silently lose bytes. 19 new tests: transient-error classifier coverage (positive + negative), retry loop (success after 2 transient failures, exhaustion on persistent failure, no-retry for permanent/auth/syntax errors), model fallback (success + one-shot cap), Codex / Gemini skip-fallback when no user model, concurrency env-var parsing (valid / invalid / out-of-range), semaphore actually caps parallelism, install-skills sentinel write + replacement notice + first-install no-notice. `KNOWN_OVERSIZED['headless_clients']` cap bumped from implicit 800 → 1100 to accommodate the ~280 LOC of new retry/semaphore/fallback infrastructure; post-v1.3.0 candidates for splitting into per-client modules. Full `uv run pytest tests/` = 931 passed.

- **Headless CLI transport hardening round 1 — UTF-8, missing-binary, auth-expiry.** Three HIGH-severity gaps in `src/coarse/headless_clients.py::_HeadlessCLIClient._run` flagged by a post-probe integration audit. (1) **UTF-8 decoding**: `subprocess.run(..., text=True, ...)` relied on `locale.getpreferredencoding(False)` for stdout decode, which is `cp1252` on default Windows and crashes with `UnicodeDecodeError` the first time a claude/codex/gemini response contains a Greek letter, math symbol, em-dash, or emoji. Now passes `encoding="utf-8", errors="replace"` to both `subprocess.run` call sites (`_run` and the `_probe_cli_help` helper). Also forces `LANG=C.UTF-8` and `LC_ALL=C.UTF-8` (or `en_US.UTF-8` on macOS) in `_clean_subprocess_env` so Node-based child CLIs write UTF-8 to stdout deterministically regardless of the user's shell locale; preserves any caller-set `LANG`/`LC_ALL` override. Windows path is a no-op since Node picks up the console codepage. (2) **Missing-binary error**: `FileNotFoundError` from `subprocess.run` when the host CLI binary isn't on PATH used to bubble up as an opaque `[Errno 2] No such file or directory: 'claude'` traceback 10 pipeline stages into a review. Now caught in `_run` and re-raised as a `RuntimeError` with a host-specific install hint: `"claude -p binary not found on PATH. Install it with: npm install -g @anthropic-ai/claude-code@latest"`. Each subclass overrides the new `_install_hint()` method so `CodexClient` points at `@openai/codex` and `GeminiClient` at `@google/gemini-cli`. (3) **Auth-expiry detection**: when the host CLI exits non-zero with "not authenticated" / "unauthorized" / "session expired" / "run `claude login`" / similar in stderr, the wrapper now classifies the error via `_classify_cli_error` and prepends a user-facing hint like `"claude -p auth failure. Run `claude login` ... to re-authenticate. Raw error: ..."`. Substring-matched (not regex) so version-drifting stderr formats still catch. Non-auth failures fall through to the existing generic `"exited N: <stderr>"` path unchanged. Nine new tests pin: UTF-8 round-trip through `_run` with Greek/math/em-dash/emoji, LANG/LC_ALL forcing on Unix + Windows skip, LANG override preservation, `FileNotFoundError` → friendly error per host, auth-pattern classifier coverage (positive + negative cases), and non-auth fallthrough. Full `uv run pytest tests/` = 912 passed.

- **Headless CLI clients now probe for version-gated flags and fall back gracefully on older installs.** Florian hit this on v1.3.0 preview: his `claude -p` version predates `--effort`, so every pipeline stage died with `error: unknown option '--effort'` and produced zero output. Fix: `ClaudeCodeClient` now probes `claude -p --help` once per class on first use, caches whether `--effort` is mentioned, and either passes the native flag (new versions) or drops it and prepends an effort-guidance block to the prompt (old versions). Same pattern applied preemptively to `CodexClient` (probes `codex exec --help` for `-c KEY=VALUE` config-override support and falls back to text injection when missing) and `GeminiClient` (probes `gemini --help` independently for `--approval-mode` — falls back to the legacy `--yolo` toggle — and `--output-format text` — drops it entirely when missing, since Gemini's default is already text). Effort-guidance text uses a single shared helper `_effort_text_prefix(effort)` so old-version fallback behavior is identical across all three clients. Probe results are cached on each class so `claude --help` runs at most once per process. Probe failures (missing binary, timeout, permission error) default to "unsupported" so the pipeline degrades gracefully instead of crashing on an unknown-flag error at stage 1. Six new tests in `tests/test_headless_clients.py` cover: Claude old-version drops `--effort` + injects text, Claude probe caches across instances, Codex old-version drops `-c model_reasoning_effort`, Gemini old-version falls back to `--yolo`, Gemini mixed version (has `--approval-mode` but not `--output-format`) picks each flag independently, and a `_reset_probe_caches` autouse fixture so test isolation is guaranteed. The existing `test_claude_effort_passes_through_unchanged` was updated to pre-set the probe cache to `True` and to assert the new flag ordering (`--output-format` before `--effort`). Full `uv run pytest tests/` = 903 passed.

- **OpenRouter file-parser 400 on large PDFs (≥ ~15 MB)** — `_extract_openrouter_file_parser` in `src/coarse/extraction_openrouter.py` base64-encoded the whole PDF inline into the JSON request body to OpenRouter. OpenRouter's request-body limit (effective ~8–16 MB on its Cloudflare Workers path) rejected anything larger with a generic HTTP 400 well before Mistral OCR ever saw the file, even though Mistral's own limits are 50 MB / 1000 pages. A real user hit this with a 20 MB paper. Fix: introduce a new `signed_url_ctx: ContextVar[str | None]` at module level — when set, `_extract_openrouter_file_parser` sends the signed URL straight through in the `file_data` field so OpenRouter fetches the PDF server-side and hands it to the backend engine without the base64 blob ever riding in the request body. The contextvar is set in two call sites: `src/coarse/cli_review.py` reads `signed_download_url` from the handoff bundle and sets the ctx around `run_headless_review` (reset in a `finally` block so no leakage across invocations), and `deploy/modal_worker.py` mints a short-lived (900 s) Supabase signed URL via `db.storage.from_("papers").create_signed_url(...)` immediately after the storage download and sets the ctx around `review_paper` (same `finally` reset). Both call sites fall back to the original base64 path when minting fails or the bundle is missing the URL, matching pre-fix behaviour exactly — a regression is impossible. The local download in `_download_handoff_source` still happens because `extraction_qa.py` needs the PDF on disk for the vision-LLM page-render check; we just stop base64-encoding it for OpenRouter. Signed URL TTL of 900 s gives the retry loop in `_post_openrouter_ocr` (9 attempts, exponential backoff, ~288 s max wall clock) roughly 3× headroom. Tests: two new tests in `tests/test_extraction_openrouter.py` cover URL mode (asserts `file_data` is the URL literal and contains no base64 prefix) and the contextvar reset path (base64 fallback restored after `.reset()`), plus the existing base64 test confirms the default path when the ctx is unset. Full `uv run pytest tests/` still green at 897 passed.

## v1.3.0 — 2026-04-14

**Major release. The big ticket items**:

- **Subscription handoff flow** — new `coarse-review --handoff <url>` command + `coarse install-skills` CLI + bundled Claude Code / Codex / Gemini CLI skill templates. Users paste one command into their coding agent and it runs the full review pipeline against their own Claude Code / Codex / Gemini CLI subscription, paying zero LLM API cost (only ~$0.15 OpenRouter extraction). Signal-driven `--attach` watcher replaces the legacy poll-based flow. Preview-first deploy workflow with Tier 0 + Tier 1 isolation docs.
- **MCP server** — new `deploy/mcp_server.py` exposes the coarse pipeline as an MCP server on Modal (`coarse-mcp` app) so Claude Desktop / Claude.ai connectors can review papers directly. Single-container session store for now (low-volume production); scaling path documented in `deploy/DEPLOY_MCP.md`.
- **Web hardening** — migrated transactional email from Gmail SMTP to Resend, Cloudflare Turnstile gate on the submit flow, fail-closed submission kill switch, session-scoped OpenRouter OAuth key storage, fenced prompt boundaries on all untrusted paper content paths, Modal webhook validation, preview-first release workflow with environment-scoped Vercel Authentication + middleware Basic Auth fallback.
- **Pipeline refactor** — extraction split by responsibility, stage helpers moved out of orchestrator, canonical pipeline stage spec, architecture anti-drift regression tests that statically scan import graphs for cycles / layer violations / oversized files.

Total: **87 commits** from 131 PRs + direct-to-dev fixes, **~21k insertions / ~2.4k deletions** across 119 files. Every line of the subscription-handoff flow is covered by the handoff-polish chain (#121-#131) documented below.

### Changed

- **Setup page: spend-limit suggestion bumped from $10 to $20, plus a new "Use my subscription" tab for the coding-agent handoff path.** Two user-reported issues:
  1. The original $10 suggestion was too close to real review cost for complex papers; a single review could exhaust the key mid-run and waste the whole spend. Step 2 (Add credits) and Step 4 (Credit limit) now both recommend $20 as the minimum, with an explanation that the cost estimate shown pre-submit is a heuristic with ~15% buffer, not a hard ceiling. The bump gives every review ~10x headroom on top of the typical $0.25–$2 cost.
  2. The setup page only covered the direct-OpenRouter flow. Users hitting "Review with my subscription" from the main form had no walkthrough of the prerequisites (install Claude Code / Codex / Gemini CLI first, log into the subscription, add minimal OpenRouter credit for the OCR pass only). Added a tab switcher (`TabSwitcher` / `OpenRouterTab` / `SubscriptionTab` in `web/src/app/setup/page.tsx`). The **OpenRouter key** tab holds the existing 5-step flow with the $20 bump. The new **Use my subscription** tab has 5 new steps: install one of the three coding agents with vendor-docs links + login/test commands per agent (`AgentCard` component), get $1 of OpenRouter credit for extraction only, upload the paper, run the three terminal commands (setup + launch + wait), and a troubleshooting section covering the three failure modes we actually saw during live testing: the "Open Claude Code" button not launching anything on Windows, the `install-skills` "no such command" graceful-degradation path, the `ANTHROPIC_API_KEY` subscription-vs-API billing footgun (fixed in v1.3.0 via `_SUBSCRIPTION_BILLING_KEYS`, but worth a note for anyone upgrading from an earlier `coarse-ink` install), and the "fewer comments than expected" section-timeout symptom. The copy is written in a direct, concrete voice with specific numbers and imperatives rather than vendor-site marketing prose. `/module-review --changed` uncovered three checklist rules that had drifted from the code during the headless-CLI / review-stages refactors, producing false-positive findings. Rule #5 previously said "`print()` is only allowed in `cli.py` top-level flow" — now correctly lists the four CLI entry-point modules that stream stdout to the user (`cli.py`, `cli_review.py`, `cli_attach.py`, `headless_review.py`), matching the split introduced when `--attach` moved out of `cli_review.py`. Rule #6 now explicitly calls out that back-compat re-export shims (`claude_code_client.py`) still need dedicated `is`-identity tests, pinning the pattern used for the new `tests/test_claude_code_client.py`. Rule #9 now carves out env-passthrough in CLI entry modules (`os.environ["OPENROUTER_API_KEY"] = ...` for subprocess inheritance, and `_find_openrouter_key`'s lightweight no-config-dep key discovery in `headless_review.py`) so the rule stops flagging intentional bootstrap code. Rule #11 was the most out-of-date: it told reviewers that individual agents had to call `quote_verify.verify_quote` before returning, but the actual design centralizes verification in `review_stages._verify_with_fallback`, which wraps every comment-producing stage (section, proof_verify, editorial, crossref, critique). The updated rule documents the wrapper as the canonical call site and flags new agents or pipeline paths that bypass it. Both `.claude/commands/module-review.md` and `.Codex/commands/module-review.md` are kept in sync so the Claude Code and Codex runs of `/module-review` apply the same rules.

### Added

- **`make release-audit` + runtime safety net for the dev → main release cutover.** The preview gate, bypass secret, preview Supabase URL, and preview Modal environment are all environment-scoped — production cold-starts never see any of it because every consumer checks `VERCEL_ENV === "preview"` first. In theory `dev → main` needs no code change. In practice the dashboard side can drift, so this commit adds three layers that make the cutover bulletproof: (1) **A new `scripts/release_audit.py`** (runnable via `make release-audit`) that greps `web/src` for any `process.env.PREVIEW_BASIC_AUTH_PASSWORD`, `process.env.PREVIEW_BASIC_AUTH_USERNAME`, or `process.env.VERCEL_AUTOMATION_BYPASS_SECRET` read that isn't within 10 lines of a `VERCEL_ENV === "preview"` guard (string literals that merely mention the variable name are skipped via a `process\.env\.` match anchor), fails exit 1 on any unguarded read, and then prints a manual dashboard checklist covering Vercel production env vars (must-have + must-not-have), Modal production secrets, Supabase migrations, and the final tag-and-push sequence. (2) **A runtime safety net** in `web/src/middleware.ts::warnIfPreviewVarsLeakedIntoProduction()` that runs once per Vercel cold start, checks whether any of the three preview-only env vars is set while `VERCEL_ENV === "production"`, and logs a loud `[release-audit]` `console.error` naming the leaked var(s) so Vercel runtime logs surface a dashboard misconfiguration that would otherwise be silent. The preview code paths themselves still no-op on production (every consumer gates on `VERCEL_ENV === "preview"` first), but the warning makes the misconfig visible instead of silent. (3) **A CI-time drift guard** in `tests/test_release_audit.py` with four tests: `test_release_audit_clean_on_dev` (asserts the current web source passes the audit), `test_release_audit_detects_injected_unguarded_read` (positive control), `test_release_audit_accepts_string_literal_mentions` (negative control), `test_release_audit_accepts_guarded_read` (negative control). The test suite pins both the detector's precision AND the current state of the web source, so any future PR that introduces an unguarded read fails CI with a pointer at the exact file:line to fix. A new **"Cutting a production release (dev → main)"** section in `deploy/PREVIEW_ENVIRONMENTS.md` explains the separation of concerns and references both the audit script and the runtime safety net. The section sits between Tier 0 and Tier 1 because it applies regardless of which preview infrastructure tier the operator has adopted.

### Removed

- **Retired the entire MCP server path.** The `coarse-mcp` Modal app, `deploy/mcp_server.py`, `deploy/mcp_test_client.py`, the `/mcp` landing page, `/api/mcp-extract`, `/api/mcp-handoff`, `scripts/run_remote_mcp_dev.py`, `tests/test_mcp_server.py`, `tests/test_remote_mcp_dev.py`, `docs/mcp-install.md`, the `mcp-ingest` CLI command in `src/coarse/cli.py`, and the `[mcp]` optional-dependency extra (`fastmcp`, `pymupdf4llm`) in `pyproject.toml` are all gone. Rationale: the MCP server was supposed to let users run a review against their Claude.ai / ChatGPT / Claude Code / Gemini CLI subscription by exposing our pipeline as MCP tools. But the architecture collapses our parallel multi-stage pipeline into a single sequential chat session inside the host LLM — no parallel section reviews (that's tool-call parallelism, not multiple Claude instances), no multi-model routing, no orchestrated editorial/critique/quote-verify stages, no cost gate, no structured output enforcement. The resulting review was strictly worse than what the real pipeline produces. Meanwhile the subscription-handoff flow (#121–#131, landed in v1.3.0) runs the **actual coarse pipeline** against `claude -p` / `codex exec` / `gemini -p` subprocesses with full parallelism, on the user's own machine, billing their subscription directly. It's the same use case — "run reviews on my subscription instead of paying OpenRouter" — and it's genuinely better. The MCP server also had a single-container bottleneck (`min_containers=1, max_containers=1` because FastMCP's session store is in-memory) that would have needed Redis or Postgres session persistence to scale. Instead of fixing that to keep a strictly worse path alive, we deleted it. What stays: `/api/mcp-finalize` is retained because it's the subscription-handoff callback (misleading name kept for migration continuity — the "mcp" in the name is historical, not architectural). `web/src/lib/mcpHandoff.ts` is retained because it holds the subscription-handoff helpers. Drift tests in `tests/test_headless_instructions.py`, `tests/test_handoff_lifetimes.py`, and `tests/test_kill_switch.py` were updated to drop the retired routes from their pinned exempt-list / route-set assertions. `scripts/release_audit.py` dashboard checklist no longer requires `MODAL_EXTRACT_URL` or `coarse-mcp` in production Modal. `deploy/DEPLOY.md` + `deploy/PREVIEW_ENVIRONMENTS.md` + `CLAUDE.md` + `AGENTS.md` got v1.3.0-retire banners at the top and the workflow files (`modal-deploy.yml`, `modal-preview-deploy.yml`) dropped the `coarse-mcp` matrix entry. **Operator action required after merging:** run `modal app stop coarse-mcp` (and `modal app stop coarse-mcp -e preview`) to tear down the deployed Modal apps — they're no longer referenced by CI but the Modal platform keeps running them until you explicitly stop them.

### Fixed

- **Pre-PR 5-agent review sweep on the release branch.** Ran `/pre-pr` on the release branch before cutting the tag; the 5 parallel review agents flagged 2 CRITICAL test gaps, 15 HIGH findings, and 23 MEDIUM items. Addressed the critical/high items that could affect end users without blocking the release:
  1. **Preview bypass secret + signed URLs scrubbed from CLI error messages.** `cli_review.py::_fetch_handoff`, `_download_handoff_source`, `_post_finalize`, and the top-level `main()` exception printer now route every URL and response-body interpolation through a new `_scrub_url()` helper that strips query strings before the URL hits stderr or the detached log file. The security-review agent caught this: `_post_finalize` was re-raising `RuntimeError(f"Callback to {callback_url} failed: ...")` and on preview deploys `callback_url` includes `?x-vercel-protection-bypass=<VERCEL_AUTOMATION_BYPASS_SECRET>` — the bypass secret was landing in the detached log file agents routinely grep. Fix preserves the scheme/host/path so the error is still useful for the human reading it.
  2. **Harmonized `signed_download_url` as the single canonical field name.** The `/h/[token]` route was emitting `signed_pdf_url` only; the CLI's `_fetch_handoff` was reading `signed_download_url` as primary with a fallback to the legacy name. In production the primary branch was always dead code and the `or`-fallback did all the real work. Tests mocked the new name — so the test suite exercised a code path that cannot fire in production. Renamed the server emission to `signed_download_url`, deleted the dead fallback and its `setdefault` cargo-cult. One name, one test path, one code path.
  3. **Deleted duplicate `_verify_with_fallback` helper.** `pipeline.py` and `review_stages.py` both defined the helper verbatim as a leftover from #100's pipeline/stage split. Canonical home is now `review_stages.py`; `pipeline.py` imports it. Every call site already passed `drop_unverified` explicitly so the default divergence didn't affect behavior, but the duplicate would silently drift on the next refactor.
  4. **`_post_finalize` retries on transient failures.** Zero-retry was losing the whole 10-25 min review on a single transient 5xx or connection error. Added a 3-attempt exponential backoff loop (2s, 4s, 8s) for connection errors, timeouts, 429, and 5xx. Permanent 4xx failures (400/401/403/404/410) still fail fast because retrying those cannot help. Token consumption is server-side atomic (`UPDATE ... WHERE consumed_at IS NULL`), so retrying on success-but-network-blipped is safe — either the first attempt persisted and the retry gets 410, or it didn't and the retry gets a fresh 200.
  5. **Handoff tempdir + fetch error-handling reshaped.** Previously `_fetch_handoff` and `_download_handoff_source` ran outside any try/except, so a stack trace from a `requests.ConnectionError` or similar would bubble up with the full signed URL in its `__str__`. And the pipeline-failure path (`return 6`) bypassed the callback-failure `finally` and leaked `handoff_tmpdir`. Restructured `main()` with a single outer try/finally wrapping from `mkdtemp` through the final return so the tempdir is cleaned up on every exit path, with catch-and-scrub around the fetch and download individually so no raw exception propagates past the CLI boundary.
  6. **`scripts/kill_switch.py` `urlopen(timeout=15)`.** Stalled Supabase edge was hanging the operator indefinitely. Interactive tool; 15s gives up fast enough.
  7. **`web/src/lib/estimateCost.ts` fetch with `AbortController(10_000)`.** Matches the pattern every other server→upstream fetch in the web surface uses (`turnstile.ts`, `mcp-extract/route.ts`, `submit/route.ts`). Without it, a hung openrouter.ai upstream would pin a Vercel function slot.
  8. **Windows `_windows_pid_alive` disambiguates the `STILL_ACTIVE == 259` collision.** Previously `GetExitCodeProcess` alone treated any process that legitimately exited with code 259 as still-running. Added a follow-up `WaitForSingleObject(handle, 0)` call — `WAIT_TIMEOUT` means still running, `WAIT_OBJECT_0` means signaled/exited. - **`coarse-review --attach <log>` — signal-driven wait mode for coding-agent handoffs.** Replaces the per-agent-call polling loop in the handoff prompt with a single blocking watch command. `coarse-review --detach` now writes `<log>.pid` next to the log file containing the subprocess PID, and the detached child registers an atexit hook to unlink its own pidfile on orderly exit. The new `--attach <log>` flag reads that pidfile, opens the log from offset 0, streams new bytes to stdout as they appear, and polls `os.kill(pid, 0)` every 5 seconds until the worker exits. Every 30 seconds of log idleness it emits a `[attach] pid=<N> elapsed=<mm:ss> — waiting…` heartbeat so Claude Code's Bash tool sees stdout activity and doesn't flag the command as hung. Exit codes map to outcome: 0 (complete), 1 (failure marker — wins over success if both present), 2 (PID dead without a completion marker — probably silent crash), 3 (pidfile missing or malformed after the 5s grace window), 124 (exceeded `--attach-timeout`, default 1800s), 130 (Ctrl+C on the watcher — detached worker keeps running, with a re-attach hint). The watcher is passive: it never signals the detached process, so the agent can safely Ctrl+C out of `--attach` and re-invoke it against the same log. `_is_pid_alive` also probes `ps -o state=` and returns false on zombie processes — a test-only concern in production (the launcher orphans the worker to init at spawn time) but critical in pytest where the test process is the parent of the fake worker. ### Fixed

- **Instructor retry errors no longer collapse to a bare `</last_exception>` tag** — `deploy/modal_worker.py::_sanitize_error` used a "last non-empty line" heuristic to trim Python tracebacks before writing `error_message` to Supabase. For `instructor.exceptions.InstructorRetryException`, whose `str(exc)` is an XML-like envelope (`<last_exception><failed_attempts><exception>…`), that heuristic kept only the outermost closing tag and threw away the actually-useful payload — so every instructor-retry failure showed up in the operator dashboard as the single string `</last_exception>`. (#62)
- **Email-capacity gate on the submit page** — when the rolling 24-hour review count crosses 240 (≈480 emails, leaving a small buffer below Gmail's free-tier 500/day sender cap), the submit form disables the email input and shows a short red note: "Daily email cap hit — save your review key to retrieve it." Submission is still allowed without an email; the Modal worker already guards `if email:` before sending the completion message, so review processing is unaffected. The threshold lives in `web/src/lib/emailCapacity.ts` (`EMAIL_CAPACITY_REVIEW_THRESHOLD`), `/api/status` returns the new `emailCapacityReached` flag alongside `activeReviews`, and `/api/submit` re-checks server-side so a client can't bypass the email regex by lying about capacity (skips the `review_emails` not-null and regex validation only when the DB confirms the gate is active, and suppresses the confirmation email when the address is empty). Motivated by the 2026-04-10/-11 traffic spike where daily volume jumped from <10 to 96-142 reviews — a single 2x growth day would have quietly started bouncing completion emails. (#59)
- **Prompt caching re-enabled for OpenRouter-routed Claude and Gemini** — `LLMClient.supports_prompt_caching` had a stale openrouter/ exclusion that silently disabled Anthropic prompt caching for every production review (coarse users upload OpenRouter keys only, never direct Anthropic). OpenRouter has forwarded explicit per-block `cache_control` breakpoints to Anthropic since mid-2024 — the gate was months out of date. Flipping it on activates two already-written caching paths: `agents/base.py::_build_messages` caches the system prompt on every agent (section agents share one system prompt across N parallel calls, so 1 write + N-1 reads fires immediately on every multi-section review), and `agents/overview.py` caches the full paper-context prefix in the system block. A follow-up commit extends the match to also cover Google Gemini models — per OpenRouter's current docs, "Gemini caching in OpenRouter requires you to insert cache_control breakpoints explicitly within message content." The match covers `anthropic/claude-*`, `openrouter/anthropic/claude-*`, `vertex_ai/claude-*`, `gemini/gemini-*`, `google/gemini-*`, `openrouter/google/gemini-*`, and `vertex_ai/gemini-*` forms; the extraction_qa vision stage (default model `gemini/gemini-3-flash-preview`) gets caching too. OpenAI and DeepSeek are deliberately NOT matched — both auto-cache prompt prefixes server-side and don't need `cache_control` blocks; adding them would clutter requests with no-op metadata. Cost math: cache writes billed at 1.25× input, reads at 0.1× (90% off), break-even at 0.28 reads per write. Every `cache_control` block in coarse's pipeline clears that bar trivially. Expected savings: ~$0.09-0.20 per sonnet review, ~$0.14-0.30 per opus review, scaling with section count and paper length. 8 new tests in `test_llm.py` cover direct anthropic, OR-routed anthropic, direct gemini, OR-routed gemini, vertex_ai/claude, vertex_ai/gemini, plus negative cases for OpenAI / DeepSeek (auto-cache, gate stays False) and undocumented providers (Qwen, Mistral, Kimi, glm, Grok).

- **`llm.py::complete()` MD_JSON floor could exceed model ceiling** — the Kimi/MD_JSON path did `clamped = max(clamped, 16384)` AFTER the initial `_clamp_max_tokens()` call, so a future Kimi variant with a smaller output window could have its max_tokens raised above what the model actually supports. Fix re-clamps after the floor. No live impact with the current Kimi k2.5 (32k output window).

Fix caps the stop at `len(paper_text) - window_size + 1` so every scored chunk is a full window.
- **`quote_verify.py::_is_math_heavy` false positives on prose with dates** — triggered strict math-mode (0.92 fuzzy threshold instead of 0.80) whenever text had three or more numeric tokens, so a prose quote like "Table 3 shows 15.2 in 2020" would flip to strict mode for the wrong reason. Fix requires a 5% density floor on top of the existing ≥3 tokens / ≥10% density condition.

## v1.1.1 — 2026-04-10

### Added

- **Web: Log in with OpenRouter** — the submission form now has a "Log in with OpenRouter" button that runs OpenRouter's browser-only OAuth PKCE flow and fills in the API key automatically. The returned key is stored in the browser's localStorage so users stay logged in across visits; a "Log out" control clears it. The manual paste field is kept as a fallback for users who prefer to use a scoped key. The server contract is unchanged — the key is still only sent in the `/api/submit` body when a review is submitted and never persisted server-side. Closes #12.

### Changed

- **Mistral OCR is now OpenRouter-only** — removed the `_extract_mistral_direct` backend that tried to hit Mistral's API directly with a `MISTRAL_API_KEY`. All PDF OCR now routes through OpenRouter's `file-parser` plugin, so users only ever need an `OPENROUTER_API_KEY`. The OCR → Docling fallback chain is unchanged for offline use.
- **Raised Modal concurrency from 10 → 20 containers**, removed the web-side hard rejection gate so bursts beyond the container limit are queued by Modal instead of erroring out.
- **OpenRouter OCR retry budget raised from 5 to 10 attempts** with a 32s per-retry backoff cap. Sequence is now 1s + 2s + 4s + 8s + 16s + 32s × 4 = 159s max, giving much more headroom for Mistral OCR's upstream timeouts while keeping the total wait bounded. A new cost guard inspects the `usage` field of every 200-with-error-body response and refuses to retry if the request was actually billed, so the retry bump can't multiply the user's API cost.
- **PDF extraction chain now has three tiers: Mistral OCR → pdf-text (OpenRouter) → Docling.** The new middle tier uses OpenRouter's free `pdf-text` file-parser engine, which extracts embedded PDF text without involving Mistral OCR at all. It catches Mistral OCR upstream outages (persistent 504 timeouts) without paying Docling's torch/RapidOCR startup cost, and falls through to Docling for scanned image-only PDFs.
- **Modal function memory raised from 2 GB to 4 GB.** Large PDFs (8+ MB) plus the Docling torch/RapidOCR stack can exceed 2 GB when OCR falls through to offline mode. 4 GB keeps the offline fallback usable for big documents.
- **PDFs are no longer deleted from Supabase Storage on the `finally` path** of the Modal worker. They're only deleted on the success path now; failed reviews keep their PDF so Modal's infrastructure retries (or a manual resubmit of the same job_id) can actually find it. Stale PDFs from failed runs are swept by the daily cleanup cron at 24h.

### Fixed

- **Docling fallback works in production** — added `libgl1` + `libglib2.0-0` to the Modal image. Without them, Docling crashed on first import with `libGL.so.1: cannot open shared object file`, meaning the "offline fallback" wasn't actually a fallback when the OpenRouter OCR path failed.
- **OpenRouter OCR error handling** — when OpenRouter returns HTTP 200 with an error body (rate limit, plugin unavailable, content policy), extraction no longer crashes with `KeyError: 'choices'`. Instead it raises a clean `ExtractionError` with the provider's error message, which the orchestrator can then classify into a user-facing message. Also handles missing `choices`, malformed annotations, empty content, and non-dict JSON responses.
- **OpenRouter OCR retry logic** — added bounded retries (3 → 5 attempts, exponential backoff 1s/2s/4s/8s/16s) on connection errors, read timeouts, and transient HTTP statuses (408, 429, 500, 502, 503, 504). Non-transient 4xx errors (401 bad key, 402 spend limit) fail fast without wasting retries. Diagnostic logging on every retry and every unexpected response shape.
- **Retry OpenRouter OCR when the error is wrapped in a 200 response body.** The file-parser plugin occasionally returns `HTTP 200` with `{"error": {"code": 504, "message": "Timed out parsing ..."}}` instead of a raw `504`. The previous retry logic only matched on transport status codes, so it treated these as immediate failures. Now retries whenever the body's `error.code` is in the same retryable set (408, 429, 500, 502, 503, 504).
- **Modal-level retries work again.** Previously, if Modal infrastructure retried a review (after an OOM or container crash), the retry would 404 on the PDF because the `finally` block had already deleted it from Supabase Storage. See the PDF storage change above.
- **Extraction QA no longer logs spurious "Page N out of range" warnings.** When Mistral OCR emitted more `<!-- PAGE BREAK -->` markers than the PDF had pages (trailing empty chunks, or a page split mid-content), `_select_qa_pages` could propose a page number that fitz rejected. Now clamps scored page numbers and the final selection set to `num_pages`.
- **Math section detection logs the underlying exception** instead of a bare "failed" message, so transient LLM errors are actually diagnosable from the Modal worker logs.
- **Math section detection no longer silently fails for Claude-family models.** The LLM call passed `max_tokens=256`, which was fine for Qwen/DeepSeek but too tight for Claude 4-family models (Opus, Sonnet) that sometimes write a prose preamble before emitting the tool call. At 256 the preamble alone could hit `finish_reason='length'` and instructor raised `InstructorRetryException("The output is incomplete due to a max_tokens length limit.")` before any JSON was produced, forcing the keyword-heuristic fallback. Bumped to 1024 and tightened `MATH_DETECTION_SYSTEM` to tell the model to skip the preamble and emit the structured response directly. Diagnosed from real user review `1e786d50`.

## v0.1.0 — 2026-03-03

Initial release.

### Features

- **PDF extraction** (`extraction.py`): Convert PDF papers to text using pymupdf4llm; optional
  vision mode for scanned or image-heavy PDFs.
- **Structure analysis** (`structure.py`): LLM-based parsing of paper into title, domain,
  taxonomy, abstract, and labelled sections with claims and definitions.
- **Overview agent** (`agents/overview.py`): Produces 4-6 macro-level issues covering the paper
  as a whole (runs in parallel with section agents).
- **Section agents** (`agents/section.py`): One agent per section produces 15-25 detailed
  comments with verbatim quotes from the paper.
- **Cross-reference agent** (`agents/crossref.py`): Deduplicates comments, validates quotes, and
  checks consistency across agents.
- **Critique agent** (`agents/critique.py`): Self-critiques the draft review and revises weak
  comments as a quality gate.
- **Pipeline** (`pipeline.py`): Orchestrates extraction, structure analysis, all agents, and
  synthesis into a single `review_paper()` call.
- **Synthesis** (`synthesis.py`): Deterministically renders a `Review` object to structured
  markdown format.
- **LLM layer** (`llm.py`): litellm wrapper with instructor for structured Pydantic output; tracks
  per-call cost.
- **Cost estimation** (`cost.py`): Pre-flight cost estimate with a configurable spending cap and
  interactive user approval gate.
- **Config management** (`config.py`): Reads and writes `~/.coarse/config.toml`; resolves API
  keys from environment variables or the config file.
- **CLI** (`cli.py`): `coarse setup` for interactive configuration; `coarse review paper.pdf` for
  reviewing a paper; `--model`, `--output`, and `--yes` flags.
- **Quality eval** (`quality.py`): Developer-only tool that scores a generated review against a
  reference review using an LLM judge.
- **Type definitions** (`types.py`): All shared Pydantic models (`PaperText`, `PaperStructure`,
  `Review`, `DetailedComment`, `CostEstimate`, etc.).
- **Prompt templates** (`prompts.py`): All LLM prompt templates in one place.
