## Overall Feedback

Here are some overall reactions to the document.

**Coding Gain Definitions and Approximation Validity**

Readers might note inconsistencies in the fundamental coding gain definition between Section I-B and Section IV-B, particularly regarding normalization and parameter relationships. Additionally, the effective coding gain rule of thumb lacks theoretical justification and specified SNR operating ranges, affecting the rigor of performance comparisons. It would be helpful to clarify the equivalence conditions for fundamental gain formulations and provide bounds or citations for the effective gain approximation.

**Lemma 5 Proof Completeness and Invariance**

Lemma 5 claims equivalence between trellis codes based on different partitions, but the proof sketch does not fully establish that code sequences remain identical under the augmented encoder transformation. Furthermore, it does not explicitly address whether decoding complexity and error coefficients are preserved under this equivalence. It would strengthen the result to provide a more rigorous demonstration of sequence invariance and specify which performance parameters remain unchanged.

**Parameter Table Consistency and Data Completeness**

Several tables exhibit inconsistencies in depth parameter notation between real and complex definitions, and contain blank entries for error coefficients in new code classes. Additionally, the distance invariance assumption for new code classes is not explicitly verified against regular labeling conditions. It would be helpful to add footnotes clarifying depth conventions, complete the missing data entries, and confirm distance invariance for the proposed classes.

**Decoding Complexity Derivation and Verification**

The normalized decoding complexity values rely on an approximation range and algorithms detailed in a companion paper, making independent verification difficult. Since complexity is a key factor in the performance comparisons, the basis for these estimates affects the validity of the conclusions. It would be helpful to provide a sketch of the complexity analysis or specific references justifying the approximation range within this document.

**Scope Alignment and Practical Performance Context**

The Abstract suggests a universal framework for coding techniques, while the body restricts analysis to lattice-type coset codes, creating a scope tension. Moreover, performance comparisons exclude shape gain, which could mislead practical rankings for implementations where constellation shaping affects power efficiency. Aligning the Abstract to reflect the lattice-specific focus and adding caveats about shape gain would improve accuracy.

**Effective Coding Gain Approximation May Not Reflect Full Distance Spectrum**

The paper assumes effective coding gain ╬│_eff can be estimated using a rule of thumb (0.2 dB reduction per factor of 2 increase in error coefficient NŌéĆ), stated in Section V. However, actual error probability depends on the full distance spectrum, not just nearest neighbors. The paper acknowledges this is 'only approximately valid for moderately low error rates' and 'generally does not take into account neighbors other than the nearest' (Section VII). This affects Tables V, VII, VIII, IX, X, and XI where ╬│_eff values are used for code comparisons. Readers might note that codes with identical ╬│_eff could have different actual performance if their higher-order distance spectra differ. It would be helpful to provide uncertainty bounds on ╬│_eff values or include additional distance spectrum parameters (NŌéü, NŌéé) for more codes beyond the Ungerboeck-type codes where these are given in Table V.

**Decoding Complexity Values Reference External Algorithms**

The paper presents decoding complexity values N_D in Tables II, III, IV, VI, VII, VIII, IX, X, and XI, but these reference 'trellis-based decoding algorithms of part II' (the companion paper). Section III states complexity is 'measured by the number of binary operations of the trellis-based decoding algorithms given in part II.' Since Part II is not included in this document, readers cannot independently verify these complexity claims or understand the specific algorithm assumptions. This affects cross-paper comparisons and implementation planning. It would be helpful to include a brief summary of the complexity calculation methodology or provide a reference to publicly available implementation specifications that would allow independent verification of the N_D values.

### 18. Trellis Code Path Distance Calculation

It would be helpful to rewrite 'is equal to $d_H(C) = 5$' as 'is 4 for the minimum weight error event' and clarify that Euclidean distance differs from bit-wise Hamming distance.
