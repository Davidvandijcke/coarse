## Overall Feedback

Here are some overall reactions to the document.

**Alignment of Abstract Claims with Scope Limitations**

The Abstract suggests that practically all known good constructive coding techniques can be characterized as coset codes, while Section I-B restricts the analysis to lattice-type coset codes. Readers might note that this discrepancy creates ambiguity regarding the generality of the classification, particularly for non-lattice group structures. It would be helpful to align the Abstract with the body text by specifying the focus on lattice-type codes or expanding the analysis to address the broader claim.

**Rigor of Effective Coding Gain Estimation**

The paper utilizes a heuristic rule of thumb regarding error coefficients to calculate effective coding gain, which is used extensively in performance tables and plots. While acknowledged as an approximation, relying on this unverified metric for primary comparisons may obscure true trade-offs without a sensitivity analysis. It would be helpful to provide a more rigorous bound on how variations in error coefficients affect performance or explicitly label these values as heuristic estimates to maintain theoretical rigor.

**Treatment of Finite Constellation and Shape Gain**

The analysis focuses on fundamental coding gain derived from infinite lattice structures, excluding shape gain which can contribute significantly in practical finite block length systems. Readers might note that dismissing shape gain could understate the performance potential of codes amenable to spherical shaping. It would be helpful to include a discussion on how constellation boundaries interact with the presented code structures to provide a more complete picture of practical utility.

**Transparency of Comparative Data and Complexity Metrics**

Key comparative tables rely on unpublished work or private communications for critical parameters like error coefficients, and decoding complexity values depend on a companion paper. This reliance limits the reproducibility of the conclusions without access to external sources. It would be helpful to include specific derivations or reference published sources where these parameters are independently verified to ensure the comparative claims are fully substantiated.

**Consistency of Dimension and Lattice Definitions**

There are potential inconsistencies in the definition of the dimension parameter $N$ across sections, and the fundamental volume definition assumes a time-zero lattice structure that may not always hold. Readers might note that ambiguity in these definitions could lead to confusion in calculating coding gains or applying the framework to non-standard codes. It would be helpful to standardize the definition of $N$ and specify the conditions under which the time-zero lattice assumption is guaranteed to be valid.

**Clarification Needed on Decoding Complexity Formula**

It would be helpful to clarify the derivation of the decoding complexity values ($N_D$) in Table IV relative to the formula provided in Section V. The text states that complexity is $eta 2^{k+r}$ where $2^{k+r}$ is the number of branches per stage. For the 16-state code ($e=4, k=2, r=1$), the text formula yields $2^{3}=8$ branches, whereas the standard trellis structure implies $16 	imes 4 = 64$ branches. The table values (e.g., 120 for the 16-state code) align more closely with the standard branch count multiplied by $eta$ (approx 112) plus a small overhead, rather than the text's explicit formula. Readers might note this discrepancy, and it would be helpful to specify whether $2^{k+r}$ refers to branches per state or total branches, or if the formula applies to the partition decoding step specifically.
